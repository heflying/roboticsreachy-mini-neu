三、sherpa-onnx RK3588 模型详情 👑

sherpa-onnx 是 唯一在 RK3588 上有官方原生支持的中文 ASR 框架，提供预编译的 RKNN 模型和完整文档。

3.1 流式模型（streaming）— 适合实时对话

模型 语言 Encoder 大小 RTF (NPU) 示例 推荐场景
Zipformer-small 中英双语 44 MB 0.20 18秒音频 → 3.5秒处理 资源敏感、低延迟优先
Zipformer 中英双语 132 MB 0.38 4.7秒音频 → 1.8秒处理 精度优先、机器人主模型
RTF < 1 即可满足实时要求。两个模型均支持麦克风实时识别（sherpa-onnx-alsa），NPU 核心可选 0~3。

3.2 离线模型（non-streaming）— 适合 VAD 分段后识别

模型 语言 最大时长 模型大小 RTF (NPU)
Paraformer-zh 中文 15 秒 ~432 MB 0.075 ⭐最快
SenseVoice 中/英/日/韩/粤 5~30 秒 ~200 MB 0.129
NPU vs CPU 速度对比（SenseVoice 10秒版）：
  1×A55 小核 CPU：RTF 0.440  |  1×A76 大核 CPU：RTF 0.100  |  1×NPU：RTF 0.129
NPU 速度接近大核 CPU，关键优势是 释放 CPU 做其他任务（如 VAD、TTS、业务逻辑）。

四、其他方案详细分析

4.1 Whisper + RKNN

部署路径：PyTorch 模型 → ONNX → RKNN 格式 → NPU 推理
优点：Rockchip 官方有 rknpu2 示例仓库，tiny/small 模型 RTF < 0.5（用 NPU + CPU 混合推理）
致命缺点：
  • 中文 CER 约 5.14%（AISHELL-1），比 Paraformer 的 1.95% 差 2.6 倍
  • 非流式（encoder-decoder 全句处理后才输出），不适合实时对话
  • 模型转换流程复杂，debug 困难

4.2 FunASR 原生部署

结论：与 sherpa-onnx 重复，无增量价值。
sherpa-onnx 已封装了 Paraformer（来自 FunASR 生态）的 ONNX 导出 + RKNN 转换，且提供了更简洁的 C++/Python API。单独部署 FunASR 意味着需要自己完成 PyTorch→ONNX→RKNN 的全流程转换和 RK3588 适配，最终效果不会超过 sherpa-onnx 已有的 Paraformer RK3588 模型。

4.3 PaddleSpeech

不推荐原因：
  • ARM64 支持主要通过 FastDeploy，但无 RK3588 NPU 加速
  • 中文流式 ASR 模型（如 DS2）精度和速度均不如 Zipformer/Paraformer
  • PaddlePaddle 框架本身在 ARM64 上的安装和依赖非常重
  • 社区在 RK3588 上几乎没有成功部署的案例

4.4 Vosk / DeepSpeech / Coqui

直接排除：
  • Vosk：有 ARM 支持但没有高质量中文模型，中文识别效果很差
  • DeepSpeech：Mozilla 已停止维护，中文模型停留在 2021 年水平
  • Coqui STT：项目已归档停止开发，同 DeepSpeech 一脉

六、结论与建议

🏆 首选方案：sherpa-onnx（唯一成熟选项）

对于「中文 + 流式 + RK3588 NPU」这个组合需求，sherpa-onnx 是唯一经过官方验证的成熟方案，没有实质性的替代品。

🔄 流式对话（机器人实时交互）

  • 首选：Zipformer-small（44MB, RTF 0.20）
  • 备选：Zipformer（132MB, RTF 0.38, 精度更高）
  • 均为中英双语，支持麦克风实时流式输入

📝 离线/高精度场景

  • 首选：Paraformer-zh（RTF 0.075, CER 1.95%）
  • 备选：SenseVoice（RTF 0.129, 多语种+情感识别）
  • 需配合 VAD 分段（如 silero-vad）
📊 最终测试矩阵
  • 评估阶段（PC CPU）：Zipformer + Paraformer，AISHELL-1 + SeniorTalk 数据集
  • 部署阶段（RK3588）：换用 sherpa-onnx RKNN 版本（sherpa-onnx-rk3588-streaming-zipformer-*），调用 NPU 加速
  • 代码几乎不用改：sherpa-onnx 的 Python API 在 x86 和 ARM64 上完全一致，只需替换模型文件路径和 provider
  • Whisper、FunASR 原生、PaddleSpeech 等方案在 RK3588 + 中文 + 流式 三个约束下均不构成有效替代
