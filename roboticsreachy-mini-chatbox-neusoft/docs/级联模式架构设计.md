# Cascade 模式架构设计

> Reachy Mini ChatBox 的级联（cascade）语音对话模式设计说明

---

## 一、概述

Cascade 模式是本项目**默认的对话模式**（在 `main.py` 中，未加 `--realtime` 时进入）。它采用经典的**级联流水线（cascade pipeline）**：

```
麦克风 → VAD → ASR → LLM → TTS → 扬声器 + 机器人动作
                  ↓
          转写实时分析（Transcript Analysis，可选反应）
```

每一阶段都是**可替换的 provider**（ASR / LLM / TTS），通过 `cascade.yaml` 配置选择，CLI 与环境变量可覆盖。这种模式与 Realtime 模式（音频到音频端到端）形成对照：本文档只描述 Cascade 模式。

性能度量体系（TTFB、各阶段热点分解）已有独立文档，见 [`级联架构性能指标设计.md`](级联架构性能指标设计.md)。

---

## 二、运行模式（三种子模式）

入口 `cascade/entry.py :: run_cascade_mode()` 根据命令行参数选择三种子模式之一：

| 子模式 | 触发条件 | 入口类 | 说明 |
|--------|---------|--------|------|
| **Gradio UI** | `--gradio` | `CascadeGradioUI` | 图形界面，含 ContinuousVADRecorder 流式录音、AudioPlaybackSystem 播放 |
| **Console（控制台）** | 默认（无 `--gradio/--autotest`） | `CascadeLocalStream` | 基于 VAD 的本地麦克风语音检测，直接驱动机器人扬声器 |
| **Autotest（自动测试）** | `--autotest` | `CascadeTestStream` | 从文本文件逐句喂入 pipeline，无需真实麦克风/扬声器 |

`entry.py` 在加载任何配置前，先把 CLI 覆盖写入环境变量（`CASCADE_ASR_PROVIDER` / `CASCADE_LLM_PROVIDER` / `CASCADE_TTS_PROVIDER`），随后初始化各类服务（movement_manager、head_wobbler、camera_worker、vision_manager），并在 Gradio 模式下调用 `handler.start()` 启动事件循环与状态机。最后统一通过 `stream_manager.launch()` 进入主循环，关闭时做资源清理并以 `os._exit(0)` 硬退出，避免 PortAudio/sounddevice 的退出竞态。

## 三、核心组件：CascadeHandler

`cascade/handler.py :: CascadeHandler` 是 cascade 模式的**总编排器**，在构造时完成：

- **Provider 初始化**：`init_asr_provider()` / `init_llm_provider()` / `init_tts_provider()`（见第四节）。
- **对话历史**：`conversation_history: list[dict]`（OpenAI Chat Completions 格式）。
- **并发控制**：`processing_lock`（asyncio.Lock），保证同一时刻只有一个 turn 在跑。
- **事件循环**：在后台守护线程中运行（`_run_event_loop`），供跨线程（如 VAD 录音线程）通过 `run_coroutine_threadsafe` 提交协程。
- **工具门控（Tool Gating）**：
  - 若 `CASCADE_DIALOG_ONLY_TOOLS` 未关闭（默认开启），则不向 LLM 暴露任何工具，**LLM 文本直接送 TTS**（`text → TTS directly`，不再依赖 `speak` 工具）。
  - `CASCADE_STREAMING_DIALOG`（默认开启）启用流式对话：LLM 边生成边流入 TTS。
  - 关闭 vision 时自动剔除 `describe_camera_image` 工具。
- **转写分析**：`transcript_manager`（`TranscriptAnalysisManager` 或 `NoOpTranscriptManager`）。
- **调度与主动引擎**：`Scheduler`（闹钟/日历）、`DecisionEngine`（主动对话决策）。
- **状态机字段**：`RobotState`（DIALOGUE / ACTIVE / RESTING）及其超时/抖动任务。

关键处理入口：
- `process_audio_manual()`：非流式（整段录音）路径，先 `asr.transcribe()` 再 `_run_pipeline_after_transcription()`。
- `process_audio_streaming_start() / _chunk() / _end()`：流式 ASR 路径，逐 chunk 拿到 partial transcript，结束取 final 后跑 pipeline。
- `_run_pipeline_after_transcription()`：校验空转写 → 重置计时器 → 开启新 turn（token/turn_id）→ 注入 deferred 提醒 → 加入用户消息 → 调用 pipeline。

## 四、Provider 系统

### 4.1 工厂（`provider_factory.py`）

`init_provider(provider_type, ...)` 按 `cascade.yaml` 选中 provider：

1. 读取 `get_<type>_provider_info()`（含 `module` / `class` / `location` / `streaming` / `requires` 等元数据）与 `get_<type>_settings()`（构造参数，剔除元数据）。
2. 注入所需 API key（从环境变量映射表取）。
3. **动态导入**模块并实例化：
   ```python
   module = importlib.import_module(f"...cascade.{provider_type}.{info['module']}")
   ProviderClass = getattr(module, info["class"])
   ```
4. 对 LLM 还会注入 `system_instructions`（会话指令 + 额外指令）；对 Spark 注入 `app_id`。

`init_transcript_analysis()` 若有 profile 的 `reactions.yaml` 则构建 `TranscriptAnalysisManager`，否则返回 `NoOpTranscriptManager`。

### 4.2 抽象基类

| 类型 | 基类 | 关键接口 | 实现示例（本 fork） |
|------|------|---------|------------------|
| ASR | `ASRProvider` | `transcribe(audio_bytes, language) -> str` | deepgram、whisper_openai、zipformer_sherpa（本地） |
| ASR（流式） | `StreamingASRProvider` | `start_stream()` / `send_audio_chunk()` / `get_partial_transcript()` / `end_stream()` / `prepare_stream()` | qwen_realtime、openai_realtime_asr 等 |
| LLM | `LLMProvider` | `generate(messages, tools, temperature, token, max_tokens) -> AsyncIterator[LLMChunk]`；可选 `warmup()`；`parse_tool_call()` | openai、gemini、ollama、spark、qwen |
| TTS | `TTSProvider` | `synthesize(text, voice) -> AsyncIterator[bytes]`；可选 `warmup()`；`sample_rate` | kokoro、kokoro_zh、piper_zh、openai、elevenlabs、gradium |

**LLM 流式协议**：`generate()` 产出 `LLMChunk`，`type` 为 `text_delta`（文本增量）/ `tool_call`（工具调用）/ `done`（完成信号）。基类 `parse_tool_call()` 默认按 OpenAI 格式解析 `(call_id, tool_name, arguments)`。

> 本 Neusoft fork 针对中文场景新增了 `zipformer_sherpa`（本地 ASR）、`piper_zh` / `kokoro_zh`（中文 TTS）、`ollama`、`spark`（科大讯飞）等 provider，`.env.example` 默认即指向这些中文 provider。

## 五、对话管线（Pipeline）

管线逻辑集中在 `cascade/pipeline.py`，由 `PipelineContext`（打包 llm / tts / speech_output / history / tool_specs / deps / result / token / turn_id）贯穿。

### 5.1 流式对话主路径 `process_streaming_dialog_response()`

```
┌─ 1. 快速回复（quick_reply）──────────────────────┐
│   命中简单话术（如"在吗"）→ 直接 speak → 结束      │
└───────────────────────────────────────────────────┘
        │ 未命中
        ▼
┌─ 2. 主循环（最多 3 次工具调用迭代）──────────────┐
│  for tool_depth in range(3):                      │
│    - LLM.generate(流式) → text_deltas 生成器       │
│    - speech_output.speak_stream(text_deltas, ...)  │
│        （LLM 边生成边切句送 TTS，见第六节）          │
│    - 聚合 text_parts / tool_calls                  │
│    - 若无 tool_calls → 记录 speak item → break     │
│    - 若有 tool_calls → execute_tool_calls() → 继续  │
└───────────────────────────────────────────────────┘
        │
        ▼
   成本归集 + 取消检查（token.cancelled）
```

- **快速回复**（`quick_reply.py`）：对高频、无状态的简短话术直接返回确定性短回复，跳过 LLM 调用以降低延迟（当前默认实现返回 `None`，接口保留）。
- **工具调用**：LLM 可返回 `tool_calls`，由 `execute_tool_calls()` 执行并把结果写回 `conversation_history`（按 OpenAI 规范，所有 tool 结果须紧跟带 tool_calls 的 assistant 消息）。
- **看图特例** `see_image_through_camera`：返回的 base64 图像不进历史（避免膨胀），改为存到 `result.captured_frames` 并以轻量 `{status, frame_index}` 标记占位；随后把图像作为 user 消息再调一次 `process_llm_response()` 让 LLM 描述。
- 非流式路径 `_process_llm_response_once()` 支持**失败重试（最多 2 次）**，重试时通过 `speech_output.speak()` 播报"稍等"，最终失败播报道歉语。

### 5.2 工具结果与多轮

- 工具执行后，若 `tool_depth < 5`，会**重新调用 LLM** 让其基于工具结果生成口语回复（streaming 路径上限为 3 次迭代）。
- 转写分析在 pipeline 跑之前并行触发（`_on_transcript_final`），**永不阻塞**主链路。

### 5.3 取消与回滚

- 每轮携带 `TurnCancellationToken`（见第七节）。若被取消：丢弃本次输出、把 `conversation_history` 回滚到 `history_checkpoint`、并追加系统消息 `[用户打断了AI的回复]`。

## 六、流式 TTS 与句子分块（SpeechOutput）

`SpeechOutput` 是 TTS 播放后端的协议，提供 `speak(text)` 与 `speak_stream(text_chunks)`。两个实现：

### 6.1 ConsoleSpeechOutput
逐 chunk 合成并通过 `playback_callback` 播放，带速率限制（按 chunk 时长 × 0.95 节流）与头部摆动（HeadWobbler）驱动。`speak_stream()` 用 `SentenceChunker` 把文本增量切成句子段后逐段 `speak()`。

### 6.2 GradioSpeechOutput（Gradio / Console 共用后端）
这是降低首音延迟（TTFB）的关键：

- **整段播报 `speak()`**：先 `split_into_sentences()` 切句，再对每句**并行**发起 TTS 合成，用一组 `gate` 事件保证句子**按序入队**（即使后发的 TTS 先返回也不乱序）。首块入队即标记 `audio_playback_started`，并触发 barge-in 监听。
- **流式播报 `speak_stream()`**：
  - `produce_segments()` 用 `SentenceChunker` 把 LLM 文本增量切成片段，`consume_segments()` 并行合成并送播放队列（generation = turn_id 用于隔离）。
  - **延迟 TTS 预连接**：在收到首个 LLM 文本增量时才触发 `prepare_stream()`，把预连接→使用的窗口从约 10s 压缩到约 0.5s，避免连接过期（详见 [`WebSocket预连接优化设计.md`](WebSocket预连接优化设计.md)）。
  - 支持 `CASCADE_RETURN_AFTER_TTS_QUEUED`：音频入队后即可返回，不等本地播放排空（进一步优化感知延迟）。

`SentenceChunker`（`streaming_text.py`）按中英文标点切分，支持 `interrupt()` —— 一旦打断，后续 `push()` 返回空、缓冲丢弃，确保中断干净。

## 七、轮次与打断管理（Turn Management & Barge-in）

为支持"用户随时插话打断机器人"，cascade 引入了逐轮（turn）生命周期与取消机制（`turn_controller.py` + `interrupt_coordinator.py`）：

- **`TurnController`**：唯一的 `turn_id`（即音频 `generation` ID）来源。
  - `start_new_turn()` → `(turn_id, token)`，audio 生成统一用 `token.turn_id` 隔离。
  - `handle_barge_in()` → 取消旧 token、中断播放（`playback.interrupt(new_generation)`）、自增 turn_id 并返回新 token。可跨线程安全调用（VAD 线程触发）。
- **`TurnCancellationToken`**：每轮独立对象，`cancelled` 为**粘性（sticky）**状态，一旦取消永久为 `True`，保证旧协程能安全终止而不影响新轮。
- **`InterruptCoordinator`**：对 LLM 任务 / TTS 消费任务 / TTS 生成器做**所有权校验**（仅当前 turn 的 token 能注册/注销），`cancel_all_for_turn()` 取消该轮所有注册资源并标记 token 取消。

数据流中的取消检查点遍布 LLM 生成、TTS 合成、播放等待；pipeline 在结束与工具执行处均会回滚历史并标记"被打断"。

## 八、对话状态机与主动能力（Proactive）

`CascadeHandler` 内维护一个扁平状态机（`_state_machine_loop`）：

```
   用户说话 / 外部触发
        │
        ▼
   ┌──────────┐
   │ DIALOGUE │  ← 初始态
   └────┬─────┘
        │ 对话结束 / 60s 无用户说话
        ▼
   ┌──────────┐                ┌──────────┐
   │  ACTIVE  │ ── 120s 无对话 ─▶│ RESTING  │
   │  中转站   │                └────┬─────┘
   └────┬─────┘                     │ 紧急闹钟触发
        │ 主动触发 (start_dialog)    │
        │                            │
        └──────────────┬─────────────┘
                       ▼
                回到 DIALOGUE

- **DIALOGUE（对话态，初始态）**：进入时重置 VAD 状态机（清掉非对话期的残留缓冲）并启动 60s 对话超时；运行 LLM pipeline；等待用户下一句或超时。VAD 仅在 DIALOGUE 态处理音频，非对话态把音频帧交给主动模块。对话结束（用户说完或 60s 超时）后回到 ACTIVE。
- **ACTIVE（活跃态，中转站）**：头部轻微摆动（wobble）+ 主动决策循环（`DecisionEngine.run`）；被闹钟/事件触发 `start_dialog=True` 时进入 DIALOGUE；120s 无对话则转 RESTING。每次对话结束都会先回到 ACTIVE，再由 ACTIVE 决定去向，因此是状态流转的中转站。
- **RESTING（休息态）**：保持静止姿态 + 主动决策循环；监听间隔更长（RESTING 用 300s 超时，ACTIVE 用 60s）；仅被紧急闹钟触发回到 DIALOGUE。

**调度与提醒**：`Scheduler`（闹钟/日历）把触发事件推入 `alert_queue`，`DecisionEngine` 在非对话态消化；进入 DIALOGUE 时把累积的 deferred 提醒格式化为系统消息注入历史，让 LLM 判断何时自然提及（紧急医疗/用药提醒不应过度延迟）。

**实现状态**：上述三态流转目前尚未完整落地。当前 `_dialogue_timeout_coro()` 将从 DIALOGUE 切回 ACTIVE 的等待时间设成了一个极大的值（`await asyncio.sleep(1000000000)`，约 31 年），实际上**禁用了 DIALOGUE→ACTIVE 的自动切换**，使机器人启动后始终停留在 DIALOGUE 态，ACTIVE / RESTING 及主动发起对话的能力暂未真正启用（函数注释仍残留"60s timeout"，与实际行为不符）。

## 九、转写实时分析（Transcript Analysis）

`transcript_analysis/` 在用户说话过程中实时触发"反应"（reactions），来源是 profile 的 `reactions.yaml`：

- **`TranscriptAnalysisManager`**：按触发类型分区构建 `KeywordAnalyzer`（关键词）与 `EntityAnalyzer`（GLiNER 实体，可选依赖）。
  - `analyze_partial()`（流式 partial，带 0.4s 去抖）与 `analyze_final()`（最终结果）均**fire-and-forget**，绝不阻塞主链路。
  - 每轮去重：非 repeatable 反应每轮最多触发一次；可重复实体反应按 `(reaction, 实体文本)` 去重。
- 无 reactions 配置时退化为 `NoOpTranscriptManager`。

handler 侧：`_on_transcript_partial()`（partial 时用 stable text 减少噪声）与 `_on_transcript_final()`（并行于 LLM）负责把文本喂给分析器，`_on_turn_complete()` 在每轮末 reset。

## 十、VAD 与句间停顿预热（Warmup）

`cascade/vad.py` 封装 Silero VAD，支持 **onnx / pytorch** 两种推理后端：

- `SileroVAD` facade 提供 `process_chunk()`，返回 `(speech_started, speech_ended, sentence_pause)` 三类事件。
- `VADStateMachine`（`LISTENING/RECORDING/PROCESSING`）负责 pre-roll 缓冲（约 0.5s）与语音检测，仅在 LISTENING 态喂音频。

**句间停顿预热（Sentence Pause Warmup）**：当 VAD 检测到说话中的静音间隙 ≥ `sentence_pause_threshold_ms` 时，handler 的 `on_sentence_pause()` 触发 `_trigger_warmup()`，用已累积的 partial 转写**提前**对 LLM 做预热（复用 KV cache）。该优化**仅对本地 LLM（location=local）有效**（云端无 cache 复用收益，自动跳过），且 warmup 失败不阻塞主流程。结束时 `_fence_pending_warmup()` 以 0ms 等待"围堵"pending warmup，避免增加用户可感知延迟。

## 十一、配置系统

### 11.1 cascade.yaml
顶层含 `asr` / `llm` / `tts` 三段（各含 `provider` 与 `providers` 字典），以及可选的 `vad`、`transcript_analysis` 段。每个 provider 定义含元数据（module / class / streaming / location / requires / hardware / install_extra）与构造参数；`CascadeConfig` 在加载时**校验** provider 是否存在、所需 API key 与依赖是否就绪、硬件是否匹配。

### 11.2 环境变量 / CLI 覆盖
| 变量 | 作用 |
|------|------|
| `CASCADE_ASR_PROVIDER` / `LLM` / `TTS` | 覆盖 `cascade.yaml` 选中的 provider（CLI 同样可传） |
| `CASCADE_VAD_BACKEND` | 覆盖 VAD 后端（onnx / pytorch） |
| `CASCADE_STREAMING_DIALOG` | 是否启用流式对话（默认开） |
| `CASCADE_DIALOG_ONLY_TOOLS` | 是否关闭工具、文本直送 TTS（默认开） |
| `CASCADE_RETURN_AFTER_TTS_QUEUED` | TTS 入队即返回，不等播放排空 |

系统提示词（`system_instructions`）由 `prompts.py` 的会话指令 + 额外指令拼成；流式对话时改用 `CASCADE_STREAMING_DIALOG_EXTRA_INSTRUCTIONS`。

## 十二、关键设计决策小结

1. **Provider 可插拔**：ASR/LLM/TTS 均为抽象基类 + 动态导入工厂，切换模型/服务商零代码改动。
2. **流式并行降低 TTFB**：LLM 生成与 TTS 合成并行；句子级分块 + 预连接延迟触发，使首音延迟显著下降。
3. **轮次隔离与粘性取消**：`turn_id` 统一隔离音频生成，`TurnCancellationToken` 粘性取消 + `InterruptCoordinator` 所有权校验，保证打断安全且不污染跨轮状态。
4. **对话/活跃/休息三态 + 主动引擎**：让机器人在无用户输入时也能基于闹钟/事件主动发起对话。
5. **文本直送 TTS（dialog-only）**：默认不暴露工具，LLM 纯文本直接播报，简化链路、降低延迟。
6. **实时分析不阻塞主链路**：转写反应与工具结果处理均为 fire-and-forget / 并行，主延迟路径只关心 ASR→LLM→TTS 关键路径（详见性能指标文档）。
7. **本地 LLM 专属优化**：句间预热仅本地生效；本地 ASR/TTS 的指标（B/D 系列）与云端区分计量。
