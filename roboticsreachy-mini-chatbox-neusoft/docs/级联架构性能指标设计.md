# 级联架构性能指标设计文档 v2.1

> Cascade 架构的统一性能度量体系（关键路径法）

---

## 一、设计理念

### 为什么分两级？

**Level 1 (一级指标)**：用户感知的延迟
- 用户问："为什么回复这么慢？"
- 答案来自 L1 指标：首音延迟 1200ms，超出预期

**Level 2 (二级指标)**：工程师定位热点
- 工程师问："慢在哪里？"
- 答案来自 L2 指标：ASR 云端处理 400ms + LLM 串行推理 600ms + Gap G1 150ms
- 进一步定位：LLM 串行推理 >500ms → 核心热点

```
用户视角 (L1)          工程师视角 (L2)
    ↓                      ↓
  "慢吗?"               "慢在哪?"
    ↓                      ↓
  TTFB=1200ms          B4=400ms, C2=600ms, G1=150ms
                           ↓
                        定位热点 → 优化 C2 (LLM推理)
```

### 关键路径法

流式级联模式下，LLM 和 TTS 并行工作：
- LLM 流式生成文字 → SentenceChunker 切句 → 首句立即送 TTS
- TTS 开始合成时，LLM 可能还在生成后续句子

因此 TTFB 由**串行关键路径**决定：

```
TTFB = B4 + G1 + C1 + C2 + D2 + D3
```

- **C2 (串行)**：`llm_stream_opened → tts_start`，LLM 对 TTFB 的直接贡献
- **C3 (并行)**：`tts_start → llm_complete`，LLM 在 TTS 已启动后的并行生成（仅监控）
- **G2 (仅监控)**：流式模式下 `llm_complete → tts_start` 可能为负值（tts_start 在 llm_complete 之前）

### B1/B2/B3 不计入 TTFB 的原因

| 指标 | 发生时机 | 为何不计入 |
|------|---------|-----------|
| B1 连接建立 | 用户说话期间 | 并行操作，预连接/复用后=0ms |
| B2 初始化 | 用户说话期间 | 并行操作，复用后=0ms |
| B3 音频上传 | 用户说话期间 | 边说话边上传，与用户说话并行 |
| **B4 云端处理** | **用户说完后** | **speech_end 后第一个串行瓶颈** |

### B5 删除说明

B5（ASR结果返回）原定义为 `asr_final_received → asr_result_delivered`。B4 已覆盖到 `asr_result_delivered`，B5 为 B4 的内部子阶段，不再单独计量。

##### 本地 ASR（如 Zipformer sherpa-onnx）⭐ v2.1 新增

| 编号 | 指标名称 | 定义 | 热点判定 | 说明 |
|------|---------|------|---------|------|
| **B1** | ASR 连接(本地) | 始终 0ms | - | 本地无需连接，标记 reuse |
| **B2** | ASR 初始化(本地) | 始终 0ms | - | 本地无会话，标记 reuse |
| **B3** | ASR chunk 解码 | 每个 chunk 解码耗时 | 仅监控 | 本地内存操作 |
| **B4** | ASR 本地处理 | end_stream→result_delivered | >100ms | **核心热点**，关键路径 |

> **自动检测逻辑**：
> - 本地 ASR：有 `asr_local_ready` 事件
> - 云端 ASR：有 `asr_ws_connected` 或 `asr_ws_reused` 事件

## 三、指标体系详解

### L1 指标（用户感知 - 与 Realtime 一致）

| 指标 | 公式 | 含义 |
|------|------|------|
| **L1 响应启动** | `response_start - speech_end` | 用户停顿后多久AI开始处理 |
| **L2 首音延迟 (TTFB)** | `first_audio - speech_end` | 用户停顿后多久听到AI声音 (**核心**) |
| **L3 转录延迟** | `transcript_show - speech_end` | 用户停顿后多久看到自己说的话 |

### L2 指标（内部分解 - 关键路径法）

#### ASR 阶段（语音转文字）

| 编号 | 指标名称 | 定义 | 热点判定 | 说明 |
|------|---------|------|---------|------|
| **B1** | ASR 连接 | WS/API 连接耗时 | >500ms | 说话期间并行，不计入 TTFB |
| **B2** | ASR 初始化 | 会话配置耗时 | >200ms | 说话期间并行，不计入 TTFB |
| **B3** | ASR 音频发送 | 音频数据上传耗时 | >200ms | 说话期间并行，不计入 TTFB |
| **B4** | ASR 云端处理 | commit→result_delivered | >300ms | **核心热点**，关键路径 |

> B5 已删除：B4 覆盖到 `asr_result_delivered`，B5 为 B4 内部子阶段

#### Gap 阶段（服务衔接）⭐ 级联特有关键

| 编号 | 指标名称 | 定义 | 说明 |
|------|---------|------|------|
| **G1** | ASR→LLM Gap | ASR完成→LLM启动 | 关键路径，>50ms 标记隐藏热点 |
| **G2** | LLM→TTS Gap | llm_complete→tts_start | **仅监控**，流式模式可能为负值 |

> **G2 设计说明**：流式级联模式下 TTS 在首句就绪时即启动（早于 llm_complete），因此 G2 = `llm_complete → tts_start` 可能为负。G2 不计入关键路径公式，仅作衔接效率监控参考。

#### LLM 阶段（推理生成）

| 编号 | 指标名称 | 定义 | 热点判定 | 说明 |
|------|---------|------|---------|------|
| **C1** | LLM 请求发送 | llm_start→llm_stream_opened | >100ms | 关键路径 |
| **C2** | LLM 串行生成 | llm_stream_opened→tts_start | >500ms | **核心热点**，关键路径串行部分 |
| **C3** | LLM 并行生成 | tts_start→llm_complete | 仅监控 | ∥并行，与 TTS 同时进行 |

> **C2/C3 设计说明**：
> - **C2 串行**：LLM 从开始到首句就绪的时间，是 TTFB 关键路径的一部分
> - **C3 并行**：LLM 在 TTS 已启动后继续生成后续句子的时间，与 TTS 并行，不影响 TTFB
> - 批处理模式下无 tts_start 时，C2 回退到 `llm_stream_opened → llm_complete`

#### TTS 阶段（语音合成）

##### 云端 TTS（如 Qwen Realtime）

| 编号 | 指标名称 | 定义 | 热点判定 | 说明 |
|------|---------|------|---------|------|
| **D1** | TTS 连接 | tts_start→tts_ws_connected | >500ms | 关键路径（reuse时=0ms） |
| **D2** | TTS 初始化 | tts_ws_connected→tts_session_update_sent | >200ms | 关键路径 |
| **D3** | TTS 首音频 | tts_commit_sent→tts_first_chunk_ready | >300ms | **核心热点** |
| **D4** | TTS 流式输出 | 完整音频生成 | 仅监控 | 依赖回复长度 |

##### 本地 TTS（如 Piper）⭐ v2.0 新增

| 编号 | 指标名称 | 定义 | 热点判定 | 说明 |
|------|---------|------|---------|------|
| **D1** | TTS 连接(本地) | 始终 0ms | - | 本地无需连接，标记 reuse |
| **D2** | TTS 线程启动 | tts_start→tts_model_generation_start | >100ms | 关键路径 |
| **D3** | TTS 模型推理 | tts_model_generation_start→tts_first_chunk_ready | >500ms | **核心热点** |
| **D4** | TTS 完整生成 | tts_model_generation_start→tts_model_generation_complete | 仅监控 | 依赖文本长度 |

> **自动检测逻辑**：
> - 云端 TTS：有 `tts_ws_connected` 或 `tts_ws_reused` 事件
> - 本地 TTS：有 `tts_model_generation_start` 但无 `tts_ws_connected`

#### 播放阶段（本地处理）

| 编号 | 指标名称 | 定义 | 热点判定 | 说明 |
|------|---------|------|---------|------|
| **E1** | 音频入队 | 音频缓冲入队列 | >50ms | 本地处理 |
| **E2** | 播放启动 | 播放设备启动 | >100ms | 本地硬件 |

### L2 指标数量

| 架构 | L2 指标数 | 说明 |
|------|----------|------|
| Realtime | 3 个 | 服务端融合，无法细分 |
| Cascade (v2.0) | 14 个 | B1-B4(4) + G1-G2(2) + C1-C3(3) + D1-D5(4,去D3) + E1-E2(2) = 15 (含本地TTS变体) |

## 四、指标关系公式（关键路径法）

### TTFB 关键路径公式

```
TTFB = B4 + G1 + C1 + C2 + D2 + D3
```

**关键路径组件**（串行，直接贡献 TTFB）：
| 组件 | 含义 | 典型值 |
|------|------|--------|
| B4 | ASR 云端处理 | 200-400ms |
| G1 | ASR→LLM 衔接 | 5-20ms |
| C1 | LLM 请求发送 | 30-80ms |
| C2 | LLM 串行生成（首句就绪） | 300-600ms |
| D2 | TTS 初始化/线程启动 | 10-50ms |
| D4 | TTS 首音频生成 | 200-500ms（云端）/ 500-800ms（本地） |

**并行监控组件**（不影响 TTFB，仅监控）：
| 组件 | 含义 | 说明 |
|------|------|------|
| B1/B2/B3 | ASR 连接/初始化/上传 | 说话期间并行 |
| C3 | LLM 并行生成 | tts_start→llm_complete |
| G2 | LLM→TTS 衔接 | 流式模式可能为负 |
| D1 | TTS 连接 | 通常 reuse=0ms |
| D5 | TTS 完整输出 | 依赖回复长度 |

### 追溯示例

```
TTFB = 1100ms
  B4 = 300ms  (ASR 云端处理)
  G1 = 10ms   (ASR→LLM 衔接)
  C1 = 50ms   (LLM 请求发送)
  C2 = 500ms  (LLM 串行生成 ← 核心热点)
  D2 = 15ms   (TTS 线程启动)
  D4 = 225ms  (TTS 模型推理)

  追溯: 300 + 10 + 50 + 500 + 15 + 225 = 1100ms ✅

  定位: C2=500ms ≥500ms → LLM串行推理是核心热点
```

## 七、关键设计决策

### 1. 关键路径法替代串行公式

**问题**: 原 L2 公式 `(B1+...+B5) + G1 + (C1+C2) + G2 + (D1+D2+D4)` 在流式模式下产生负值

**分析**:
- 流式级联下 LLM 和 TTS 并行工作
- G2 = `llm_complete → tts_start` 为负值（tts_start 更早）
- C2 覆盖到 llm_complete 包含了与 TTS 并行的时间

**解决**: 采用关键路径法
- C2 截断到 `tts_start`：只计算 LLM 对 TTFB 的串行贡献
- C3 标记为 `∥并行`：LLM 在 TTS 启动后继续生成的时间
- G2 改为仅监控：流式模式下不参与公式计算

### 2. B1/B2/B3 不计入 TTFB

**分析**: 这三个指标发生在用户说话期间，与用户说话并行执行
- B1 连接：预连接或首次建连，通常在首句音频到达时完成
- B2 初始化：连接建好后立即配置
- B3 音频上传：边说话边上传

**结论**: 这些是并行操作，不增加用户说完话后的等待时间

### 3. B5 删除

**分析**: B5 定义为 `asr_final_received → asr_result_delivered`
- B4 已覆盖到 `asr_result_delivered`
- B5 是 B4 的内部子阶段，时间已被包含

**结论**: 删除 B5，减少指标冗余

### 4. 本地 TTS 自动检测

**问题**: Piper 本地 TTS 产生不同事件名，云端 TTS 指标全显示 N/A

**解决**: 自动检测 TTS 类型
- 云端：有 `tts_ws_connected` 或 `tts_ws_reused`
- 本地：有 `tts_model_generation_start` 但无 WebSocket 事件
- 本地 D1 始终为 0ms（无连接），D2/D4 使用不同事件区间

### 5. 流式输出不评级

**问题**: C3/D5 流式输出时间依赖回复长度

**解决**: 仅监控显示，不进行 Excellent/Good 评级

### 6. 预连接显示方式

**问题**: 预连接的 WebSocket reuse 如何体现？

**解决**:
- 首次连接：显示实际耗时
- Reuse：显示 0ms 并标注 `(reuse)`

## 九、D1 指标细化说明

### D1 定义更新

D1（TTS连接建立）的计算逻辑随预连接优化方案更新：

| 场景 | D1 值 | 说明 |
|------|-------|------|
| reuse 成功 | **0ms** | 预连接已完成且未过期，直接 reuse |
| 等待成功 | **0ms** | 预连接进行中，等待完成后 reuse |
| 等待超时 | ~200ms | 等待 500ms 超时后重新连接 |
| stale 重连 | ~200ms | 预连接过期，清理后重新连接 |
| 首次连接 | ~200-500ms | 无预连接，首次建立 WebSocket |
| 本地 TTS | **0ms** | 无需连接，始终标记 reuse |
