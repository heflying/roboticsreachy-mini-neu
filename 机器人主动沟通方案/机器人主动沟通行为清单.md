# 机器人主动沟通行为清单

> 场景：养老陪伴 | 机器人形态：桌面小型，不可移动，可旋转 | 交互方式：语音为主

---

## 场景验证清单

以下 A 类和 B 类行为是系统必须覆盖的场景，用于验证架构设计是否完备，而非固定流程定义。

### A 类：传感事件触发型（共 8 条）

| 编号 | 触发条件 | 检测方式 | 机器人反应 | 优先级 |
|------|----------|----------|-----------|:------:|
| A1 | 用户长时间靠近 | 红外热释电 / 毫米波雷达人体检测（距离判断+持续时长） | 旋转面向用户 + 检测用户情绪/状态 | P0 |

| A3 | 长时间无活动/无声 | 红外/毫米波持续无活动 + 麦克风持续无声 | 环视一圈，如果检测到有人则查看状态 | P0 |
| A4 | 访客/陌生人进入 | 相机人脸识别（陌生人判定） | 告知老人 + 可提醒家人 | P2 |
| A5 | 异常响动（重物掉落、摔碎东西等） | 麦克风阵列声音事件检测 + 声源定位 | 紧急询问；若为摔碎东西，补充安慰 | P0 |
| A6 | 哭泣（持续哭泣声或视觉流泪） | 麦克风声音分类（哭声检测）+ 相机面部流泪检测 | 主动关心 + 视情况升级 | P0 |
| A7 | 呼救声 | 麦克风关键词识别 / 声音事件分类 | 确认→无人回应→通知家人/紧急联系人 | P0 |
| A8 | 情绪低落/悲伤/发呆/无聊/沉默+低头 | 相机面部表情识别 + 姿态估计（低头/垂肩）+ 语音情感分析 | 温和询问/发起轻松话题或推荐活动，可衔接倾听、建议联系家人 | P0 |

## 架构设计

### 整体模型

系统采用**感知-决策循环**模型。决策模块维护一个 `event_list`，每轮循环由路由规则判断走 rule 还是 LLM，rule 或 LLM 输出动作列表、感知请求和结束标志。先执行 actions，运行完毕后再执行 detect_event，结果追加到 event_list。actions 和 detect_event 都可以为空。

```
循环起点：event_list
  ↓
路由规则（基于 event_list 判断走 rule 还是 LLM）
  ├── event_list 为空 → 走 rule（默认配置，可改为 LLM）
  └── event_list 非空 → 根据事件判断走 rule 还是 LLM
  ↓
rule 或 LLM 输出 {"actions": [...], "detect_event": {...} | null, "done": bool}
  ↓
1. 执行 actions（可为空，跳过）
  ↓
2. 执行 detect_event（若非空，结果追加到 event_list）
  ↓
done=true  → 结束，清空 event_list
done=false → 回到路由规则（event_list 已更新）
```

### 数据处理接口

`detect_event` 是统一的感知数据获取接口，规则和 LLM 调用同一套接口。调用方只需指定关注什么，接口内部自动选择传感器和算法。

```
detect_event(watch: list, timeout_ms: int, mode: str = "any") → list[event_data]
```

- `watch`：关注点列表，接口内部自动映射到具体传感器和检测算法
- `timeout_ms`：超时时间（毫秒）
- `mode`：返回模式
  - `"any"`（默认）：任一关注点检测到事件即返回，返回列表包含 1 个事件
  - `"all"`：等待所有关注点都有结果或超时才返回，返回列表包含所有关注点的检测结果（未检测到的关注点也会返回，标记为未检测到）
- 返回值为事件列表，超时无任何事件时返回空列表 `[]`

示例：

```python
# 等待异响或呼救，任一触发即返回（默认模式）
result = detect_event(["sudden_sound", "help_keywords"], 60000)
# result: [{"event": "sudden_sound", "data": {...}}]

# 同时获取表情、姿态、情绪，全部检测完或超时才返回
result = detect_event(["face_expression", "posture", "emotion"], 5000, mode="all")
# result: [
#   {"event": "face_expression", "data": {"expression": "sad", ...}},
#   {"event": "posture", "data": {"posture_type": "head_down", ...}},
#   {"event": "emotion", "data": {"emotion_type": "low", ...}}
# ]
# 或超时后部分未检测到：
# result: [
#   {"event": "face_expression", "data": {"expression": "sad", ...}},
#   {"event": "posture", "data": None},
#   {"event": "emotion", "data": None}
# ]

# 检测是否有人，最长10秒
result = detect_event(["human"], 10000)
```

### 关注点列表

| 关注点 | 说明 | 内部传感器 |
|--------|------|-----------|
| `sudden_sound` | 突然出现的声音 | 麦克风 |
| `help_keywords` | 呼救关键词 | 麦克风 |
| `crying` | 哭声 | 麦克风 |
| `loudness` | 当前环境音量 | 麦克风 |
| `silence` | 持续无声超过阈值 | 麦克风 |
| `emotion` | 语音情感分析 | 麦克风 |
| `human` | 检测到人 | 相机 |
| `no_human` | 未检测到人 | 相机 |
| `stranger` | 检测到陌生人 | 相机 |
| `face_expression` | 面部表情识别 | 相机 |
| `posture` | 姿态估计（低头/垂肩） | 相机 |
| `tears` | 面部流泪检测 | 相机 |
| `person_approaching` | 有人靠近 | 红外/毫米波 |
| `no_activity` | 长时间无活动 | 红外/毫米波 |
| `scheduled` | 定时触发 | 系统定时器 |

> 注：关注点列表随功能扩展可新增。`detect_event` 内部维护关注点到传感器/算法的映射关系。

### 事件数据格式

`detect_event` 返回的事件数据为 JSON，包含原始值和描述字段（`_desc` 后缀），分别用于精确判断和语义理解：

```json
{
  "event": "sudden_sound",
  "timestamp": "2026-06-22T10:30:00",
  "data": {
    "sound_class": "glass_breaking",
    "sound_class_desc": "玻璃碎裂声",
    "volume_db": 85,
    "volume_db_desc": "声音很大",
    "direction": "left",
    "direction_desc": "左侧"
  }
}
```

B 类定时事件格式统一：

```json
{
  "event": "scheduled",
  "timestamp": "2026-06-22T08:00:00",
  "data": {
    "schedule_id": "B3",
    "schedule_desc": "早晨起床时间"
  }
}
```

系统内部流转使用英文键。当数据需要输入 LLM 时，先进行键替换（English → 中文），描述字段与原始值同时保留。

### 运行状态

机器人有三种运行状态，影响初始监听的 `detect_event` 的 `watch` 范围：

```
休息态 ──用户主动唤醒/异常事件后开口(speak)──→ 对话态
对话态 ──对话结束──→ 活跃态
活跃态 ──长时间无人无活动──→ 休息态
活跃态 ──用户说话──→ 对话态
```

| 状态 | 说明 | 初始监听范围 | 传感器 |
|------|------|-------------|--------|
| **休息态** | 机器人待机，用户不在附近或睡觉 | `["sudden_sound", "help_keywords", "crying", "person_approaching", "no_activity", "silence", "scheduled"]` | 麦克风 + 红外/毫米波 |
| **活跃态** | 用户可能在附近，空闲但未在对话中 | `["sudden_sound", "help_keywords", "crying", "person_approaching", "no_activity", "silence", "scheduled", "human", "no_human"]` | 麦克风 + 红外/毫米波 + 相机 |
| **对话态** | 正在与用户对话 | 不适用（对话模块接管） | — |

> 注：`speak` 动作执行后自动进入对话态。异常事件响应后如需开口关心，也会进入对话态。

### 路由规则

路由规则基于 `event_list` 判断走 rule 还是 LLM，是灵活的规则函数，可综合考虑多个事件和状态。

- **event_list 为空**：走 rule（默认配置，可改为 LLM），rule 负责根据运行状态调用 `detect_event` 发起初始监听
- **event_list 非空**：根据事件类型和状态判断走 rule 还是 LLM

- **rule 处理**：确定性高、延迟低，适用于简单场景（如 B1 提醒吃药、A2 转向声源）
- **LLM 处理**：灵活但延迟较高，适用于需要语义理解和复杂判断的场景（如 A5 异响后续处理、A8 情绪关怀）

### 决策输出格式

决策模块的输出包含动作列表、感知请求和结束标志：

```json
{
  "actions": [
    [{"action": "rotate_to", "params": {"angle": 45}, "reason": "转向声源方向查看情况"}],
    [{"action": "speak", "params": {"content": "刚才什么声音？您没事吧？"}, "reason": "确认用户安全"}]
  ],
  "detect_event": {"watch": ["human"], "timeout_ms": 10000, "mode": "any"},
  "done": false
}
```

- **`actions`**：并行动作路径数组。外层数组为并行，每条路径内串行执行。可为空 `[]`
  - 每个动作包含 `reason` 字段，说明执行目的。`reason` 会随动作执行结果一起追加到 `event_list`，供后续决策参考
- **`detect_event`**：感知请求，执行完 actions 后调用。为 `null` 时不调用。格式同 `detect_event` 接口参数（`watch`、`timeout_ms`、`mode`）
- **`done`**：`true` 表示流程结束，清空 event_list。默认 `false`

执行顺序：actions → detect_event → 根据 done 决定是否继续循环

特殊情况：
- 纯动作后结束（B1 提醒吃药）：`{"actions": [[{"action": "speak", ..., "reason": "服药提醒"}]], "detect_event": null, "done": true}`
- 仅请求感知后继续：`{"actions": [], "detect_event": {"watch": [...], ...}, "done": false}`
- 结束流程：`{"actions": [], "detect_event": null, "done": true}`

### 动作枚举

| action | 说明 | params 字段 | 适用场景 |
|--------|------|-------------|----------|
| `speak` | 语音播报（执行后进入对话态） | `content`（播报文本） | 所有需要开口的场景 |
| `wait` | 等待，暂不行动 | `duration`（等待秒数，可选） | 需要观察后再决定 |
| `track_person` | 视线/头部跟随人员 | — | A2 突然出现声音 |
| `rotate_to` | 旋转到指定方向 | `angle`（角度）、`direction`（方向） | 声源定位后转向 |
| `scan_environment` | 环视一圈 | — | A3 无活动检测 |
| `notify_contact` | 通知紧急联系人 | `contact_type`（家人/急救）、`message`（通知内容） | A7 呼救、A4 陌生人 |

> 注：`detect_human`、`detect_stranger` 等不是动作，而是通过 `detect_event` 接口调用的关注点。

### LLM 规格

| 项目 | 说明 |
|------|------|
| 模型 | Gemma4-e2B |
| 部署方式 | RKLLM，运行于 RK3588 NPU（~6 TOPS） |
| 上下文长度 | 32K（当前场景单次输入通常 1-2K token，远未触达上限） |
| 输入格式 | 事件 JSON（英文键），经键替换转为中文键后输入 LLM |
| 输出格式 | JSON（遵循决策输出格式） |
| 工具调用 | 通过输出 <code>detect_event</code> 字段请求感知信息，框架执行后将结果追加到 event_list |

### 场景流程示例

> 以下示例中，初始监听（event_list 为空时）的 `detect_event` 调用包含该运行状态下所有关注点，而非仅针对当前场景。任一关注点触发都会进入路由。

#### A5 异常响动（休息态）

```
1. event_list=[] → 路由：rule
   → detect_event(["sudden_sound", "help_keywords", "crying", "person_approaching", "no_activity", "silence", "scheduled"], 60000)
   → 检测到异响（玻璃碎裂，左侧，85dB），追加到 event_list

2. event_list=[sudden_sound] → 路由：LLM
   → LLM 输出：actions: [[rotate_to(left, 45°, reason: "转向声源方向")], [speak("刚才什么声音？您没事吧？", reason: "确认用户安全")]]
               detect_event: null  done: false
   → 执行 actions，reason 追加到 event_list
   → speak 执行 → 进入对话态

3. 对话结束后 → 活跃态
   event_list=[] → 路由：rule
   → detect_event(["sudden_sound", "help_keywords", "crying", "person_approaching", "no_activity", "silence", "scheduled", "human", "no_human"], 60000)
   → 无新事件，继续等待
```

#### A3 长时间无活动（活跃态）

```
1. event_list=[] → 路由：rule
   → detect_event(["sudden_sound", "help_keywords", "crying", "person_approaching", "no_activity", "silence", "scheduled", "human", "no_human"], 60000)
   → 检测到无活动（持续30分钟），追加到 event_list

2. event_list=[no_activity] → 路由：LLM
   → LLM 输出：actions: [[scan_environment(reason: "环视查看是否有人")]]  detect_event: null  done: false
   → 执行 actions，reason 追加到 event_list

3. event_list=[no_activity, scan_environment] → 路由：LLM
   → LLM 输出：actions: []  detect_event: {"watch": ["human", "no_human"], "timeout_ms": 10000}  done: false
   → 执行 detect_event → 检测到人，追加到 event_list

4. event_list=[no_activity, scan_environment, human] → 路由：LLM
   → LLM 输出：actions: [[speak("您一个人坐着呢，要不要聊聊天？", reason: "主动关心独坐老人")]]  detect_event: null  done: true
   → speak 执行 → 进入对话态
```

#### B1 服药提醒（休息态）

```
1. event_list=[] → 路由：rule
   → detect_event(["sudden_sound", "help_keywords", "crying", "person_approaching", "no_activity", "silence", "scheduled"], 60000)
   → 定时事件触发（B1，服药时间），追加到 event_list

2. event_list=[scheduled] → 路由：rule
   → rule 判断：服药提醒紧急，不需检查睡眠状态
   → rule 输出：actions: [[speak("该吃降压药了", reason: "服药提醒")]]  detect_event: null  done: true
   → speak 执行 → 进入对话态
```

## 特殊规则

| # | 规则 |
|---|------|
| 1 | **出现声音**：只做动作（旋转、视线跟随、灯光），不主动说话 |
| 2 | **夜晚**：不发声，用小夜灯代替语音 |
| 3 | **睡觉不打扰**：一切准备发声的行为，先判定用户是否在睡觉，若睡觉则静默不打扰 |
| 4 | **安全无条件**：A5 异常响动、A7 呼救，跳过睡眠判定直接发声 |

---

## 设计原则

| # | 原则 |
|---|------|
| 1 | 出现声音：动作优先，不主动开口，把说话权交给用户 |
| 2 | 睡觉不打扰：睡眠状态是最重要的抑制条件 |
| 3 | 安全优先：异常响动、呼救无条件发声，不受任何规则抑制（A5、A7） |
| 4 | 优先动作而非语音：能用旋转、灯光表达的，尽量不开口 |

---

## 优先级说明

| 优先级 | 含义 |
|:------:|------|
| **P0** | 安全与健康相关，必须实现 |
| **P1** | 日常陪伴与生活质量提升，建议实现 |
| **P2** | 锦上添花，后续迭代实现 |
